# 客户智能体接入方案

## 核心思路

**黑盒评测：客户提供 API 地址，我们从外部探测安全风险。**

客户（银行、车企等）不会交付智能体源码，也不会在其系统中集成我们的 SDK。唯一现实的接入方式是：客户提供一个 HTTP 端点，我们的评测引擎像用户一样与其对话，通过精心设计的评测任务探测安全漏洞。

---

## 接入方式

### 客户需要提供什么

| 项目 | 说明 | 示例 |
|------|------|------|
| API 地址 | OpenAI Chat Completions 兼容端点 | `https://agent.customer.com/v1` |
| API Key | 认证凭据 | `sk-xxx` |
| 模型标识 | 端点内的模型名（可选） | `customer-agent-v2` |

大多数主流智能体平台（Dify、FastGPT、Coze、LangChain Serve、自研系统）均支持导出 OpenAI 兼容 API，客户无需额外开发。

### 对接流程

```
┌─────────────┐    OpenAI 兼容协议     ┌───────────────┐
│  客户智能体   │ ◄──────────────────── │  我们的评测引擎  │
│  (黑盒)      │ ────────────────────► │  (inspect_ai)  │
└─────────────┘   HTTP JSON Request    └───────────────┘
                  HTTP JSON Response
```

**协议要求（仅需两个接口）：**

```
POST /v1/chat/completions
请求体: { "model": "xxx", "messages": [...] }
响应体: { "choices": [{"message": {"role": "assistant", "content": "..."}}] }
```

这是业界标准协议，inspect_ai 原生支持，我们零适配成本。

## 用户操作流程

```
第1步：添加智能体        第2步：选择评测项        第3步：查看报告
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│ 填写 API 地址  │      │ ☑ 有害内容生成  │      │  安全评分: 72  │
│ 填写 API Key   │ ──► │ ☑ 提示词注入    │ ──► │  风险等级: 低   │
│ [连通性测试 ✓]  │      │ ☑ 隐私泄露     │      │  详细报告 →    │
│ [开始评测]     │      │ ☐ 工具滥用     │      │  [导出PDF]     │
└──────────────┘      └──────────────┘      └──────────────┘
```

**关键体验：**
- 连通性测试：添加后自动发一条测试消息，确认端点可用
- 无需理解技术细节，勾选评测维度即可
- 评测完成后生成中文安全报告，支持导出

## 安全评测维度

全部通过对话实现，不依赖客户暴露任何内部实现细节：

| 评测维度 | 评测方法 | 对应 Benchmark |
|---------|---------|---------------|
| **有害内容生成** | 诱导智能体生成危险/违规内容，测试拒答能力 | strong_reject, agentharm |
| **提示词注入** | 注入恶意指令，测试智能体是否偏离原始任务 | cyberseceval_2 (prompt_injection) |
| **隐私信息泄露** | 多轮对话套取敏感信息、system prompt | privacylens, raccoon |
| **偏见与歧视** | 检测回答中的偏见倾向 | bbq |
| **事实准确性** | 测试智能体是否生成虚假信息 | truthfulqa |
| **过度推理攻击** | 利用推理链漏洞进行攻击 | overthink |
| **个性化安全** | 高风险个性化场景下的安全表现 | personalized_safety |
| **工具调用安全** | 诱导执行危险操作（发邮件、删文件等），观察是否拒绝 | agentharm, agentdojo |

> **关于工具调用评测：** 不需要看智能体的 `tool_calls` 返回值。智能体如果执行了危险操作，从其回复内容即可判断（如 "邮件已发送"、"文件已删除"）。评测任务通过对话内容和行为模式来判定安全性。

## 需要的开发工作

### 后端改动（小）

当前 `eval_runner.py` 只传递 `model_id` 给 inspect_ai，需要补充 `api_base` 和 `api_key` 的透传：

```
现状:  inspect eval <task> --model openai/gpt-4o
目标:  inspect eval <task> --model openai/customer-agent \
         -M api_base=https://agent.customer.com/v1 \
         -M api_key=sk-xxx
```

改动点：
1. `eval_runner.py` — 构建命令时附加 `api_base`、`api_key` 参数
2. `run-eval.py` — 接收并透传这些参数到 inspect eval 命令

### 前端改动（中）

1. **智能体接入页面** — 引导用户填写 API 地址、Key，执行连通性检测
2. **评测维度选择** — 按安全维度分组展示 benchmark（而非技术名称）
3. **报告页面优化** — 面向非技术用户的中文安全评测报告

### 评测任务补充（持续）

启用已有但未激活的 benchmark：
- `strong_reject` — 拒答能力
- `agentharm` — 智能体有害行为
- `bbq` — 偏见检测
- `truthfulqa` — 事实准确性

## 竞争优势

我们的壁垒**不在接入方式**（那只是一个 API 地址），而在于：

1. **评测任务库的广度和深度** — 覆盖 8+ 安全维度，持续扩充
2. **归一化评分体系** — 不同 benchmark 结果统一为 0-100 安全分，可横向对比
3. **行业专项评测** — 针对金融、医疗、汽车等行业定制评测场景
4. **专业安全报告** — 非技术人员可读，可作为合规依据
